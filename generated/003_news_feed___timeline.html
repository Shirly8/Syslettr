Systems Design · Tier 1

News Feed / Timeline

A news feed looks simple — show posts from people you follow — but the challenge explodes when a celebrity with 50 million followers posts something and every follower's feed must update within seconds. Interviews use this problem to probe fan-out architecture, the push vs. pull tradeoff, and how to handle hotspot users without melting your write path.

Interview

Interviewer
We're building a news feed similar to Twitter or Facebook — users follow other users, and they see posts from people they follow in reverse-chronological order. Where do you start?
You
Before diving in — what's the scale? DAU (daily active users) in the millions or hundreds of millions? And are we purely reverse-chronological, or is there a ranking/ML layer? Also, do we need to support celebrities — users with millions of followers — or is the follow graph roughly symmetric?
Interviewer
300 million DAU, reverse-chronological only for now. Assume a power-law follow graph — some users have tens of millions of followers. Feed must load under 200ms p99 (99th percentile — the slow tail, not the average).
You
Okay. 300M DAU — assuming each user opens the feed ~5 times a day, that's 1.5B feed reads/day, roughly 17,000 reads/sec at peak. Writes: if 5% of users post once a day, that's 15M posts/day, ~175 posts/sec raw — but fan-out (sending each post to all followers' feeds) explodes that to potentially billions of writes/sec for celebrities. That asymmetry is the core problem. I'll design a hybrid push-pull strategy: push to normal users' precomputed feeds, pull on-demand for celebrity posts.

Core Components

Interviewer"Walk me through what happens the moment a user hits 'post' — what does that write path look like?"
Post Ingestion Service
  • WHY: Without an async write path, a celebrity posting would block the HTTP response for seconds while fan-out completes.
  • HOW: The client sends the post to a Post Service, which writes the post to a Posts DB and immediately publishes an event to a message queue (a durable buffer that decouples producers from consumers). The HTTP response returns instantly with the post ID — fan-out happens asynchronously downstream. I store the raw post once; feeds only store references (post IDs), not copies.
  • TRADEOFF: Async fan-out means followers don't see the post in their feed the instant it's published — there's a short propagation delay, typically under a second for normal users.
Interviewer"How do you handle a celebrity with 50 million followers posting — you can't fan-out to 50 million feed caches synchronously, right?"
Feed Fan-out Worker (Push vs. Pull Hybrid)
  • WHY: Naive push fan-out for a celebrity post means 50M cache writes triggered by a single event — this saturates workers and adds seconds of latency for followers.
  • HOW: I split users into two tiers. For normal users (< ~10K followers), fan-out workers read the author's follower list and push the post ID into each follower's feed cache (a precomputed, paginated list of post IDs stored in Redis). For celebrity users (> 10K followers — threshold is tunable), I skip precomputed fan-out entirely. Instead, at feed read time, the Feed Read Service fetches the user's precomputed feed and merges it with a live pull of the last N posts from each celebrity they follow. This keeps write amplification (the multiplier from one write triggering many downstream writes) bounded.
  • TRADEOFF: Hybrid adds complexity — the read path must merge two sources, and the celebrity threshold needs tuning per deployment.
Interviewer"When a user opens their app, what does the read path look like end to end?"
Feed Read Service
  • WHY: The feed read is the latency-critical path — missing the 200ms p99 SLA here is what users feel directly.
  • HOW: The Feed Read Service checks Redis for the user's precomputed feed (a sorted set of post IDs ordered by timestamp). If present, it fetches those post IDs and does a batch lookup on a Post Cache (another Redis layer) for the actual post content. For any celebrity posts the user follows, it pulls the latest N posts from the Posts DB and merges them by timestamp in memory before returning the final page. Pagination uses cursor-based pagination (stateless — client sends the last-seen post ID, not a page number) to avoid the N+1 offset problem.
  • TRADEOFF: Merging celebrity posts at read time adds a synchronous DB query to the hot path — mitigated by a small per-celebrity post cache with a short TTL (time-to-live — how long a cache entry stays valid).
Interviewer"Fan-out workers need to look up all followers for a given user — how do you store and serve that efficiently?"
Follow Graph Store
  • WHY: Fan-out workers need to enumerate millions of follower IDs per celebrity post; a naive relational join on a single DB instance is too slow.
  • HOW: I store the follow graph in a graph-friendly key-value structure — specifically Redis sorted sets keyed by user ID, mapping to follower IDs. For the celebrity tier, follower lists are too large for Redis memory, so I store them in a separate sharded (data split across machines by user ID) follow DB (e.g. a wide-column store). Fan-out workers stream follower IDs in batches of 1,000 and enqueue sub-tasks per batch rather than loading the full list into memory at once.
  • TRADEOFF: Keeping a separate follow store means follow/unfollow operations must update both the graph DB and invalidate any cached fan-out metadata — two writes that must be eventually consistent.

Architecture

Bottlenecks

Interviewer"What would break in this design at scale?"
1A celebrity with 50M followers posts — fan-out workers must write 50M feed cache entries, overwhelming the write throughput of Redis.
Fix: I skip cache fan-out entirely for celebrity authors (above the follower threshold). Instead, I maintain a separate 'celebrity post' cache keyed by author ID with a short TTL. At read time, the Feed Read Service fetches from both the user's precomputed feed cache and the celebrity post cache for each celebrity they follow, merging results in memory. This moves the fan-out cost off the write path and distributes it across millions of readers instead of concentrating it in the write workers.
Tradeoff: Every feed read now has a small overhead proportional to how many celebrities a user follows — if a user follows 100 celebrities, that's 100 small cache lookups at read time.
2Feed cache misses on cold start or cache eviction cause every read to hit the Posts DB directly, creating a read thunderstorm (many simultaneous DB queries from cache misses).
Fix: I use a cache-aside pattern with eager warming: when a user logs in, a background job pre-populates their feed cache by fetching the last 200 post IDs from their follow graph. I also set a probabilistic early expiration (re-computing the cache entry slightly before it expires, not after) to avoid the thundering herd problem (all cache entries expiring simultaneously under traffic). Additionally, the Posts DB is read-replicated — read queries go to replicas (copies that sync from the primary), so even a full cache miss distributes load.
Tradeoff: Eager warming on login adds latency to the login path and wastes compute if the user doesn't actually scroll their feed.
3The follow graph lookup for large accounts — iterating 50M follower IDs — is slow and blocks fan-out workers from processing the next event.
Fix: Fan-out workers stream follower IDs from the Follow Graph DB in cursor-paginated batches of 1,000 rather than loading the full list. Each batch enqueues a sub-task back onto the queue. This keeps per-worker memory bounded and allows horizontal scaling — 100 workers each processing 1,000-follower batches in parallel completes 100K fan-outs per second per worker group.
Tradeoff: Batched fan-out for large accounts means the last follower sees their feed updated slightly later than the first — up to a few seconds for 50M followers at 100K fan-outs/sec.

Reference

API
POST/api/v1/posts— Create a post; body: {text, media_ids[]}; returns post_id and server timestamp immediately — fan-out is async
GET/api/v1/feed?cursor={post_id}&limit={n}— Fetch paginated feed; cursor is last-seen post ID for stateless pagination; returns [{post_id, author_id, content, created_at}]
POST/api/v1/follow/{target_user_id}— Follow a user; idempotent; triggers async follow-graph update and optional feed backfill for recent posts
DELETE/api/v1/follow/{target_user_id}— Unfollow a user; removes from follow graph and triggers async pruning of target's posts from follower's feed cache
GET/api/v1/posts/{post_id}— Fetch a single post by ID; served from Post Content Cache with Posts DB fallback
Database & Storage
  • Posts DB — PostgreSQL sharded by author_id; append-heavy workload fits relational model; wide-column would also work but SQL joins simplify backfill queries
  • Schema: posts(post_id UUID PK, author_id UUID, content TEXT, media_ids UUID[], created_at TIMESTAMPTZ, is_deleted BOOL)
  • Index: (author_id, created_at DESC) composite index for efficient 'latest N posts by author' — the hot query pattern for celebrity pull at read time
  • Sharding key is author_id not post_id — co-locates all posts by a user on one shard, making the author-feed pull a single-node query; tradeoff is shard hotspots for celebrity authors (mitigated by read replicas per shard)
  • Follow Graph DB — wide-column store (e.g. Cassandra-style) keyed by (follower_id, followee_id); optimized for streaming 'all followers of X' scans that fan-out workers need
  • Feed Cache — Redis sorted set per user_id, scored by post timestamp; ZREVRANGE gives latest N post IDs in O(log N); evict LRU users after 7 days of inactivity to bound memory
  • Post Content Cache — Redis hash per post_id with 24-hour TTL; populated on first read; reduces Posts DB QPS (queries per second) by ~90% for trending posts
