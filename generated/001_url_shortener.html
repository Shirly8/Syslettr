URL Shortener

A URL shortener looks trivial until you reason about generating billions of unique codes without collisions at high write throughput while serving redirects under 10ms. Interviewers use it to probe ID generation strategies, caching discipline, and how candidates think about read/write asymmetry.

Interview

Interviewer
We want to build a URL shortening service like bit.ly. Where do you start?
You
Before diving in — what's the target scale? Millions of links created per day or billions total? Is redirect latency SLA-bound? And do users get custom aliases or is everything auto-generated?
Interviewer
100 million new links per day, 10 billion redirects per day. Redirects must complete in under 50ms p99. Custom aliases are a nice-to-have, core is auto-generated codes.
You
So roughly 1,160 writes/sec and 115,000 reads/sec — about 100:1 read-heavy. That means caching (keeping hot URL mappings in memory close to servers) is the central scaling lever. For storage: a short code is ~7 bytes, a long URL averages ~100 bytes, plus metadata — call it 200 bytes per record. 100M records/day × 365 × 5 years × 200 bytes ≈ 36TB over five years, manageable on a single sharded (data split across multiple DB machines) cluster. I'll target 7-character base-62 codes (62^7 ≈ 3.5 trillion combinations), which gives us headroom well beyond any realistic write volume.

Core Components

Interviewer"How do you generate the short codes at 1,160 writes/sec without duplicates or coordination bottlenecks?"
Short Code ID Generation
  • WHY: Naive random generation causes collisions at scale and requires expensive DB uniqueness checks on every write.
  • HOW: I'd use a distributed ID service that pre-allocates ranges — each app server checks out a block of 10,000 IDs from a central counter stored in a DB row with atomic increment. The server converts the numeric ID to base-62 (a–z, A–Z, 0–9) locally, producing a 7-character code with no coordination per request. If custom aliases are needed, I store them in the same table with a unique index and let the DB enforce uniqueness only for that path.
  • TRADEOFF: Pre-allocated ranges mean if a server crashes it wastes up to 10,000 IDs — that's cosmetically non-sequential but functionally harmless given 3.5 trillion total codes.
Interviewer"Redirects need sub-50ms p99 — how do you actually hit that at 115K reads/sec?"
Redirect Cache Layer
  • WHY: A DB lookup on every redirect at 115K rps would require an impractically large and expensive DB fleet, and latency would blow past 50ms under any real load.
  • HOW: I put an in-memory cache (Redis or a similar distributed key-value store) in front of the DB. Short code is the key, long URL is the value. With an LRU eviction policy (drops least-recently-used entries when cache is full), a modest 50GB cache holds ~500 million entries. URL access follows a power-law (a small fraction of links get the vast majority of clicks), so a cache hit rate above 95% is realistic, meaning fewer than 6K rps actually reach the DB. On a cache miss, the app server fetches from DB, writes back to cache, and redirects.
  • TRADEOFF: Stale cache entries can survive up to the TTL (time-to-live — max age before a cached entry expires) after a URL is deleted or updated, so deletes need an explicit cache invalidation call.
Interviewer"Walk me through exactly what happens when a user submits a long URL."
Write Path & Storage
  • WHY: Without a clear write path, you end up with race conditions where two servers could theoretically generate the same code.
  • HOW: Client POSTs the long URL to any app server behind a load balancer. The app server claims its next ID from its local pre-allocated range (or fetches a new range if exhausted), encodes to base-62 locally, then writes a single row to the primary DB with the short code, long URL, owner ID, expiry, and created-at timestamp. No uniqueness check needed for auto-generated codes — the range allocation guarantees uniqueness. The response returns the short URL immediately.
  • TRADEOFF: The primary DB is a single write bottleneck; I'd shard (partition) by the first character of the short code to distribute writes across 62 logical shards if write volume grows.
Interviewer"Do you track click data, and how do you do it without adding latency to the redirect?"
Analytics & Click Tracking
  • WHY: Writing analytics synchronously on every redirect would add DB write latency directly to the user-facing redirect path.
  • HOW: On redirect, the app server fires a write to a message queue (a durable buffer decoupling producers from consumers) asynchronously — it does not wait for the ack before returning the 302. A separate consumer service drains the queue and batch-writes click events (timestamp, short code, referrer, country derived from IP) to a columnar (column-oriented, fast for aggregations) analytics store.
  • TRADEOFF: Analytics lag behind real time by seconds to minutes depending on queue depth — acceptable for dashboards, not for real-time abuse detection.

Architecture

Bottlenecks

Interviewer"What would break in this design at scale?"
1A single Redis cluster becomes a hot spot: top 1% of short codes (viral links) get hammered by millions of concurrent reads, saturating network bandwidth on a small number of cache nodes.
Fix: I'd add a local in-process cache (a small HashMap inside each app server's memory, capped at ~1,000 entries per instance) in front of Redis. Viral URLs get cached right at the process level — they never leave the machine. The local cache has a very short TTL (5–10 seconds) to bound staleness. This eliminates the Redis round-trip entirely for the hottest keys, which represent the majority of traffic.
Tradeoff: A URL deletion or update won't propagate to local caches for up to 10 seconds, so there's a brief window where deleted links still resolve.
2The primary URL DB becomes a write bottleneck if a single machine handles all 1,160 writes/sec plus replication overhead.
Fix: I'd shard the URL table by short code — hash the first character (base-62 gives 62 natural buckets) to route each write and read to the correct shard. Each shard is an independent primary-replica pair. App servers maintain a shard map (a lookup table of code-range → DB host) in config so they route locally without a coordinator. This spreads both write load and storage linearly.
Tradeoff: Cross-shard queries (e.g., 'list all links for a user') become expensive because you'd have to fan out to all shards — I'd solve that by keeping a separate user → short_codes index table on a different store.
3Cold start after a deploy flushes the Redis cache: the DB gets hit with the full 115K rps until the cache warms up, potentially causing an outage.
Fix: On deploy, I'd implement cache warming by pre-loading the top-N most-accessed short codes (queryable from the analytics store, ordered by click count in the last 24 hours) into Redis before routing live traffic to new instances. This is a simple batch job that runs as part of the deploy pipeline. I'd also use a rolling deploy (taking down one instance at a time) so the remaining warm instances absorb traffic while new ones are loading.
Tradeoff: Pre-warming adds 1–3 minutes to deploy time and requires a reliable signal for 'top URLs', which lags real-time spikes by at least the analytics batch interval.

Reference

API
POST/api/v1/links— Body: {long_url, custom_alias?, ttl_days?}. Returns {short_url, code, expires_at}. Creates a new shortened link.
GET/{code}— Public redirect endpoint. Returns HTTP 302 with Location header set to the original URL, or 404 if expired/not found.
DELETE/api/v1/links/{code}— Auth-required. Soft-deletes the link, sets deleted_at, triggers cache invalidation for that code.
GET/api/v1/links/{code}/stats— Returns click count, referrer breakdown, and country distribution for the given short code.
Database & Storage
  • PostgreSQL (or MySQL) — strong consistency on writes, mature sharding support, row-level locking for safe atomic counter increments.
  • Schema: urls(code VARCHAR(7) PK, long_url TEXT NOT NULL, owner_id BIGINT, created_at TIMESTAMPTZ, expires_at TIMESTAMPTZ, deleted_at TIMESTAMPTZ)
  • Index: PRIMARY KEY on code (B-tree, supports O(log n) redirect lookup); secondary index on owner_id for user link listings.
  • Tradeoff: TEXT for long_url is uncapped but adds variable-length storage overhead; capping at 2,048 chars covers 99.9% of real URLs and keeps row size predictable.
