Systems Design · Tier 1

Rate Limiter

Rate limiting sounds simple — count requests and reject above a threshold — but coordinating counts across hundreds of servers without a central bottleneck or race conditions is genuinely hard. Interviewers use it to probe your understanding of distributed consistency, latency budgets, and graceful degradation under load.

Interview

Interviewer
We need to build a rate limiter for our public API. Where do you start?
You
Before diving in — are we rate limiting per user, per API key, or both? What's the scale — thousands of services or a global API with hundreds of millions of users? And what's the failure mode preference: drop requests the moment we're uncertain, or allow occasional over-limit traffic to avoid false rejections?
Interviewer
Per API key, globally enforced. We have roughly 50 million active API keys and expect up to 1 million requests per second across our fleet. We'd rather drop a few legitimate requests than allow a bad actor to burst through.
You
Got it — so roughly 1M RPS (requests per second) total, 50M keys. Average limit is probably something like 1,000 req/min per key, but a small set of enterprise keys might have 100K/min. I'll assume limits are configurable per key and stored centrally. The hard problem is enforcing a global count across, say, 200 API server nodes without each node needing a synchronous round-trip to a single counter store on every request — that would add 5–20ms per call and create a single point of failure. I'll design around local approximate counting with a fast shared store for coordination.

Core Components

Interviewer"Why token bucket over a simple fixed window counter?"
Token Bucket Algorithm
  • WHY: Fixed windows allow 2x the limit at window boundaries — a client sending 1,000 req at 11:59 and 1,000 at 12:00 passes both checks but fires 2,000 in two seconds.
  • HOW: Each API key gets a bucket with capacity N and a refill rate of N tokens per 60 seconds. Each request atomically decrements the token count; if it hits zero, we reject. I store the token count and last-refill timestamp together in a single Redis entry and use a Lua script (a small script run atomically inside Redis itself, preventing race conditions) to check-and-decrement atomically.
  • TRADEOFF: Clients can still burst up to the full bucket size in one instant — which is intentional but must match the stated SLA.
Interviewer"You mentioned avoiding a round-trip to Redis on every request — how do you actually do that?"
Local Counter Cache with Redis Sync
  • WHY: At 1M RPS across 200 nodes, each node handles ~5,000 RPS; a synchronous Redis call per request adds serialized latency and makes Redis the bottleneck.
  • HOW: Each API server node keeps an in-memory local counter per key. On every request, it decrements locally. Every 100ms (configurable), the node pushes its local delta (the accumulated count since last sync) to Redis via INCRBY and reads back the global count. If the global count exceeds the limit, the node switches that key to 'reject mode' locally until the next sync window. This gives us eventual consistency — we might allow a small burst over the limit — but keeps Redis write load at 1/500th of request rate.
  • TRADEOFF: We can over-serve by up to (num_nodes × local_window_count) requests before global enforcement catches up — acceptable if we set limits conservatively.
Interviewer"Where do the per-key limits actually live, and how do nodes get updated limits in real time?"
Rate Limit Config Service
  • WHY: Hard-coding limits or requiring deploys to change them is operationally unacceptable; enterprise customers need limit changes within seconds.
  • HOW: Limits are stored in a relational DB (the source of truth) keyed by API key ID. A config service reads this DB and publishes limit changes to a pub/sub channel (a broadcast mechanism where subscribers receive updates as they happen). Each API server node subscribes to this channel and holds limits in a local hash map with a 60-second TTL (time-to-live — expiry after which it re-fetches) as a fallback.
  • TRADEOFF: There's a short propagation lag — up to a few seconds — before all nodes see a new limit; we accept this for operational simplicity.
Interviewer"What does the client actually see when they're rate limited, and how do they know when to retry?"
Rejection Response & Headers
  • WHY: Without retry guidance, rate-limited clients tend to hammer the API immediately, amplifying the problem.
  • HOW: On rejection, I return HTTP 429 with three headers: X-RateLimit-Limit (the key's configured cap), X-RateLimit-Remaining (tokens left in the current window), and Retry-After (seconds until the bucket refills enough for one request). These values come from the last Redis sync so they're accurate within the sync window.
  • TRADEOFF: Retry-After is approximate — it's based on the last-sync state, not a real-time read — but it's close enough and avoids an extra Redis call on the hot rejection path.

Architecture

Bottlenecks

Interviewer"What would break in this design at scale?"
1Redis becomes a hot spot if all 200 nodes sync every 100ms for 50M keys — that's up to 2M ops/sec against a single logical store.
Fix: Shard (split) Redis by API key using consistent hashing (a scheme where each key maps deterministically to one shard, and re-sharding moves minimal keys). Each API key always goes to one Redis shard, so no cross-shard coordination is needed. I'd run 20 Redis shards — each handling 100K ops/sec, well within Redis's single-node ceiling of ~500K ops/sec. Each API node maintains shard-aware routing using a client-side ring.
Tradeoff: Adding or removing shards requires a careful rehashing window during which some keys temporarily double-count — I'd handle this with a brief read-from-both-shards transition period.
2During a Redis node failure, the keys on that shard become unreachable and we must decide: fail open (allow all traffic) or fail closed (block all traffic for those keys).
Fix: I'd configure Redis in cluster mode with one replica per shard. On primary failure, the replica is promoted in under 10 seconds via Redis Sentinel (a watchdog process that monitors Redis and triggers automatic failover). During that 10-second window, I'd fail open — nodes fall back to local-only counting. The worst case is a small burst above limit for one window, which is acceptable given our stated preference was avoiding false rejections over strict enforcement.
Tradeoff: The fail-open window means a targeted attack could briefly exceed limits during failover — acceptable for most APIs but not for financial or abuse-sensitive endpoints.
3A single high-traffic API key (e.g. a large enterprise customer) creates a hot key in Redis — one shard gets hammered while others sit idle.
Fix: For keys with limits above 10K req/min, I split the counter across M sub-keys (e.g. key:0 through key:9), routing each request to a random sub-key and multiplying the per-sub-key limit by 1/M. On sync, I sum all sub-key values to get the true global count. This is counter sharding (spreading one logical counter across multiple physical keys to distribute load).
Tradeoff: Summing M sub-keys on each sync adds M Redis reads per hot key — still cheap, but the consistency window widens slightly because sub-keys update independently.

Reference

API
POST/api/v1/rate-limits/check— Internal endpoint called by API nodes; body: {api_key, endpoint}; returns {allowed: bool, remaining: int, retry_after_ms: int}
GET/api/v1/rate-limits/{api_key}— Admin endpoint — returns current token count, limit config, and last-sync timestamp for a given key
PUT/api/v1/rate-limits/{api_key}/config— Updates the limit for a key; triggers a pub/sub broadcast so all nodes refresh within seconds
DELETE/api/v1/rate-limits/{api_key}/counter— Admin reset — clears the Redis token bucket for a key (e.g. after a false positive ban)
Database & Storage
  • Redis (cluster mode) — primary store for token bucket counters; sub-millisecond reads/writes, TTL-based auto-cleanup.
  • Schema (Redis hash per key): HSET rl:{api_key} tokens INT remaining_ms INT last_sync TIMESTAMP — all fields updated atomically via Lua.
  • PostgreSQL — source of truth for limit configs; Schema: api_key_limits(api_key VARCHAR PK, limit_per_min INT, burst_cap INT, tier VARCHAR, updated_at TIMESTAMP).
  • Index: PRIMARY KEY on api_key for O(1) config lookups; secondary index on tier for bulk updates (e.g. downgrade all 'free' tier keys at once).
  • Key tradeoff: Redis holds no durable state — if the entire cluster is lost, counters reset to full; this is acceptable because limits are a sliding window and clients get a grace burst, but it means a total Redis failure briefly removes rate limiting.
